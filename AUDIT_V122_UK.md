# AUDIT v122 — Firebase limit optimization + private profile security

Дата: 2026-06-10

## Що змінено

### 1. Адмінка більше не читає всю колекцію `users` при відкритті

У `js/pages/admin-page.js` вкладка **Гравці** тепер використовує нову функцію:

- `listRegisteredUsersPage()`

Замість повного читання:

```js
getDocs(collection(db, 'users'))
```

адмінка читає сторінку через Firestore cursor pagination:

- `orderBy('createdAt', 'desc')`
- `limit(10 + 1)`
- `startAfter(lastDoc)` для кнопки **Вперед**

Це означає, що при 4000 гравців вкладка **Гравці** не повинна брати ~4000 reads за одне відкриття. Вона бере приблизно 11 reads на сторінку без фільтрів.

### 2. Ферми не рахуються як 10 основних гравців

Пагінація працює по основних документах `users/{uid}`.

Якщо показано 10 основних гравців і в них є ферми, на екрані може бути більше рядків, але Firestore reads все одно йдуть по 10 основних документах, бо ферми зберігаються всередині профілю користувача.

### 3. Адмінська статистика бере цифри з public JSON cache

Верхні цифри адмінки тепер намагаються братися з:

```text
public-cache/stats-summary.json
```

Це не витрачає Firestore reads. Якщо cache недоступний, адмінка показує fallback по поточній сторінці.

### 4. Регіони в адмінці завантажуються ліниво

Раніше `loadAdminData()` одразу читав регіони. У v122 регіони читаються тільки коли користувач відкриває вкладку **Регіони**.

Також в адмінці використовується:

```js
listRegionCatalog({ includeInactive: true, skipPublicPlayers: true })
```

Це не читає всю колекцію `publicPlayers` для побудови списку регіонів.

### 5. Security audit завантажується ліниво

`security-page.js` більше не запускає перевірку одразу при відкритті `admin.html`.

Перевірка запускається тільки коли відкрито вкладку **Безпека** або коли сторінка `security.html` відкрита напряму.

### 6. Заявки на ролі обмежені

`listRoleRequests()` тепер читає максимум 50 pending-заявок за раз і рахує reads у локальному лічильнику.

### 7. Правила безпеки для `users/{uid}` посилені

У `firebase/firestore.rules` змінено `canReadUser(userId)`.

Тепер повний приватний профіль `users/{uid}`, де може бути email, можуть читати тільки:

- сам власник профілю;
- Owner;
- Admin;
- Moderator.

Consul і Officer більше не мають читати `users/{uid}`. Для них мають використовуватись публічні/регіональні документи без email:

- `publicPlayers/{uid}`;
- `regions/{region}/players/{uid}`;
- D1/cache snapshots.

## Очікувана економія

При 4000 гравців:

- було: відкриття адмінки могло брати тисячі Firestore reads;
- стало: вкладка **Гравці** бере приблизно 11 reads на першу сторінку без фільтрів.

Фільтри можуть читати більше, але запит обмежений без повного сканування всієї колекції.

## Важливо перевірити після деплою

1. Увійти як Admin і Moderator.
2. Відкрити `admin.html` → **Гравці**.
3. Перевірити, що спочатку показує 10 найновіших основних гравців.
4. Натиснути **Вперед** і **Назад**.
5. Увімкнути **Основи + ферми** — ферми мають іти під основним гравцем.
6. Відкрити **Регіони** — регіони мають завантажитись тільки після відкриття вкладки.
7. Відкрити **Безпека** — перевірка має стартувати тільки після відкриття вкладки.
8. Перевірити Firestore rules у Firebase Console.

## Не зроблено автоматично

- Не створено Firebase composite indexes. Якщо Firebase покаже помилку індексу для фільтрів, потрібно натиснути посилання з Firebase Console і створити індекс.
- Не перенесено всі сторінки повністю на D1. v122 головно закриває найбільшу проблему: адмінка і приватні профілі.
